Two layers of "on"

The master switch governs everything the Spoon does — rename, picker, popups, hotkeys, sync. Auto-sync governs only the watcher: when off, the rest of the Spoon still works, you just sync displays manually via Sync Now.

Why two display sections?

"Detected Displays" answers what's plugged in right now — pure reference, regenerated on open. "Sync Groups" is the editable assignment. UUID lives in the inventory only, where it disambiguates identical models.

Groups are implicit

There's no separate "list of groups" — only a list of displays and what each is assigned to. Groups come into existence when a display picks + New group and dissolve when the last member leaves. No "create empty group" affordance, no orphans.

The single-group constraint

Each row's selector picks one group or None. The data model is a groupOf map keyed by position, so overlap is impossible by construction — not a runtime check.

Stale entries

Position 5 is in SpacesSync.json but no display sits there. Surfaced as a warning instead of being silently dropped — leave it (will activate on replug) or remove it.

Apply persists; pathwatcher hot-reloads

Apply writes ~/.hammerspoon/SpacesSync.json and re-runs the spoon's setup. Hand-editing the JSON triggers the same path via hs.pathwatcher — both surfaces converge on the same code.

Tooltips, not docs

Each timing field has a hover-info tooltip that explains the why, not just the what. The numbers exist to compensate for specific macOS behaviors — the user shouldn't have to read source comments to know why 0.3s and not 0.1s.

Hotkeys live with the rest of config

Bindings persist in SpacesSync.json alongside groups and timing — same hand-edit + GUI surfaces, same hot-reload via hs.pathwatcher. Sync Now keeps its inline kbd hint on the control panel; the Hotkeys section is where the actual binding is changed. Click a cell to re-record, Esc to cancel.
